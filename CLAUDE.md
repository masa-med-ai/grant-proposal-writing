# CLAUDE.md｜科研費 基盤C・若手研究 調書作成PJT

> このファイルは本ディレクトリで作業する Claude への**最優先指示**です。
> ユーザーのメッセージを処理する前に、必ず本ファイルを読み、ここに書かれたルールに従ってください。

---

## 0. このPJTの目的

科研費（**基盤C** または **若手研究**）の研究計画調書（研究目的・研究方法、本文4ページ）を、
4ステップのワークフローで**中断・再開可能な形で**作成する。

---

## 1. 起動時の初動（毎セッション必須）

ユーザーからの最初の依頼を受けたら、**作業前に以下の順で必ず読む**：

1. `./CLAUDE.md`（本ファイル）
2. `./01_進行ガイド.md`（マスターコントロール、全Stepの詳細手順）
3. `./02_状態管理.json`（`current_step` と `completed_steps`、`notes` を確認）
4. `./persona.md`（**本フォルダ内に存在すれば必ず Read**。ユーザーの所属・専門・実績・保有データ等の参照用）
5. 親ディレクトリの `about-me.md`, `brand-voice.md`（存在すれば。Personaとトーンの参照用）
6. 直近で更新された `90_成果物/decisions_log.md`（既に決定済みの事項）

そのうえで、ユーザーに **「現在 Step○ の途中です。続きから進めますか？」** と確認してから動く。
新規依頼であっても、本PJT内の依頼であれば必ず `01_進行ガイド.md` のフローに沿わせる。

## 2. 全Step共通の遵守ルール

| # | ルール |
|---|--------|
| R1 | **作業前に `02_状態管理.json` を Read**。`current_step` と完了済Stepを把握。 |
| R2 | **作業後に `02_状態管理.json` を Update**（`last_updated` / `current_step` / `completed_steps` / `notes`）。 |
| R3 | 中間成果物は **対応する Step フォルダ配下に保存**（既存ファイルへの上書き可）。 |
| R4 | 重要決定（テーマ・研究期間・マイルストーン等）は `90_成果物/decisions_log.md` に追記。 |
| R5 | PubMed 検索は **MeSH で検索式立案 → `count` で件数検証 → `search`** の順。 |
| R6 | PubMed 引用は **1st author / 雑誌名 / 発刊年 + ハイパーリンク** で記載（ユーザー設定）。 |
| R7 | 各 Step の DoD（完了基準）を満たしたら、ユーザーに **「次のStepに進んでよいか」を必ず確認**。 |
| R8 | Step3 の本文ドラフト段階では **Word ファイルを作成しない**（最終調整はユーザー側）。 |
| R9 | ユーザーが「再開して」「現状は？」と言ったら、`02_状態管理.json` を読んで要約提示 → 合意後に再開。 |
| R10 | 評定要素（学術的「問い」/ 独自性・創造性 / 着想の経緯 / 国内外動向と位置づけ 等）は **本文に埋没させず、必ず `##` 小見出しとして独立**させる。 |
| R11 | **想定読者は「博士号レベルだが門外漢の専門家」**。一般常識で通じる用語以外は **括弧書きで短く言い換え**を添える（例：「○○（前がん病変のひとつ）」「ベイズ最適化（少ない試行で最適パラメータを探す統計手法）」）。 |
| R12 | **過度な英語・カタカナ語の連発を避ける**。1段落あたりカタカナ専門用語は **概ね3語まで**を目安とし、初出時は日本語訳を併記する。略語は初出で必ずフルスペル＋日本語訳を示す（例：「人工知能（AI; Artificial Intelligence）」「ランダム化比較試験（RCT）」）。少数の英単語は許容するが、文章のリズムが英語に支配されないようにする。 |
| R13 | **著者本人の原著論文（自己引用）を本文中にさりげなく織り込む**こと。Step1-4 でテーマ確定後、PubMed MCP を用いて **第一著者・有名誌・テーマと親和性の高い論文** を候補として抽出し、ユーザー合意の上で `13_テーマ候補メモ.md` の「自己引用候補」セクションに登録する。Step3 では「着想に至った経緯」「予備データ」「研究実績」など自然な箇所に **1〜3 本** を埋め込む（自慢調にならないよう、淡々と先行研究として参照する）。 |

## 4. Step ごとの要点（詳細は `01_進行ガイド.md`）

### Step1：テーマ壁打ち＋環境整備
- **Visualizer（`mcp__visualize__show_widget`）でチェックリストUIを提示**し、4項目（Persona / KAKEN MCP / PubMed MCP / テーマ壁打ち）を可視化する。
- KAKEN MCP / PubMed MCP の有無を確認。無ければユーザーに導入を案内（KAKEN: https://github.com/masa-med-ai/KAKENHI-search-mcpbundle ）。
- テーマ壁打ちは **3〜5 回のラリー**。実現可能性・新規性・規模感を毎回評価。
- **完了条件**：`13_テーマ候補メモ.md` 冒頭の `FINAL_THEME` に1テーマが書かれ、ユーザー合意済み。

### Step2：研究計画の具体化
- 研究期間：若手 2〜4 年 / 基盤C 3〜5 年。
- 研究項目は 3〜6 個に分解。
- マイルストーンは **数値化必須**（「○年○月までに○例登録完了」など）。抽象表現は禁止。
- `23_ガントチャート.md` に **Mermaid `gantt` 記法**で記述しユーザーに表示・合意取得。

### Step3：本文ドラフト執筆
- 最初に「若手研究 or 基盤C」を確認 → 該当の `31_*` または `32_*` の注意事項を**必ず Read してから**執筆。
- 評定要素は独立採点されるため、**必ず以下を `##` 小見出しとして立てる**：
  - 学術的「問い」
  - 学術的独自性・創造性
  - 着想に至った経緯
  - 国内外の研究動向と本研究の位置づけ
  - 研究目的を達成するための研究方法・研究計画
  - 研究目的・研究方法の妥当性・独自性・創造性
- 文字量目安：**4ページ＝合計 5,000〜5,500 字以内**（1ページ約 1,200〜1,400 字）。
- 出力は `.md` のみ。**Word ファイルは作らない**。
- **平易化チェック（R11/R12）**：ドラフト完成後に「専門用語に短い言い換えが付いているか」「カタカナ語が密集していないか」を必ず1パス見直す。
- **自己引用チェック（R13）**：`13_テーマ候補メモ.md` の「自己引用候補」を再読し、最低1本は本文に織り込まれていることを確認する。引用は **1st author / 雑誌名 / 発刊年 + ハイパーリンク** で記載。

### Step4：ポンチ絵プロンプト生成
- FINAL_THEME と研究方法に基づき 1〜2 案を提案。各案について構図・要素・配色・メッセージを1段落で。
- `41_ポンチ絵プロンプト.md` に **ChatGPT (DALL·E / GPT-image) 用（英語・詳細）** と **NanoBanana 用（簡潔・構図重視）** の両方を出力。

## 5. 使用が推奨されるスキル / MCP

| 場面 | 推奨ツール |
|------|-----------|
| Step1 のチェックリストUI表示 | `mcp__visualize__show_widget`（read_me → show_widget の順） |
| KAKEN 類似研究調査 | `mcp__KAKEN_Search__*`（未導入なら導入案内） |
| PubMed 文献調査 | `mcp__pubmed__count` → `mcp__pubmed__search` → `mcp__pubmed__fetch` |
| ガントチャート描画 | Mermaid `gantt` 記法（`23_ガントチャート.md` に直接記述） |
| 進捗管理 | `TaskCreate` / `TaskUpdate`（各Stepの実行時に立てる） |
| Persona参照 | 本フォルダの `persona.md`（あれば）、および親ディレクトリの `about-me.md`, `brand-voice.md` を Read |

---

## 6. ふるまいのトーン

- **研究者ユーザーへの対応**として、フォーマルかつ簡潔に。
- 不明点はユーザーに質問する前に、まず本ファイル・進行ガイド・状態管理を読んで自己解決を試みる。
- Step の DoD を満たさないまま次に進まない。
- 「とりあえず書いて出す」のではなく、各 Step で**ユーザー合意を取りに行く**ことを優先する。

---

## 7. 困ったときの優先順位

1. `01_進行ガイド.md` を再読する
2. `02_状態管理.json` の `notes` を読む
3. `90_成果物/decisions_log.md` で過去の決定を確認する
4. それでも不明ならユーザーに `AskUserQuestion` で確認する
